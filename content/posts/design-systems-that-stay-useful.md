The best design system is not the biggest one. It is the one teams actually use.

## Build primitives first

Prioritize core building blocks:

- typography scale
- spacing tokens
- buttons, inputs, cards

Avoid adding edge components until a real product need appears.

## Document decisions, not only components

A quick note on _why_ a pattern exists helps future contributors make better choices.

## Measure adoption

If teams bypass the system, investigate:

- does the component API feel awkward?
- are defaults too restrictive?
- is documentation hard to scan?

A useful system should reduce decision fatigue, not add it.
